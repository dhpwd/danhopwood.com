- **The journey – four attempts (hook material):**
  - **Umso (manual GUI):** evaluated alongside Webflow and Framer; Umso was the fastest of the three to get something live. "Fastest GUI" still means every piece of copy, every image, every section layout is constructed by hand. Scaling = manual labour per page. Death by a thousand clicks
  - **Astro port of the Umso export (older repo exists separately):** the intent was right – get off GUI, into code, make it AI-maintainable. The execution was agonising. Umso's exported HTML and JS is generated output, not hand-written code, and reverse-engineering it into a clean Astro project took an unreasonable amount of time. Lesson: you can't port a generated codebase into a maintainable one. Start clean
  - **Lovable (React SPA):** currently live at fidero.com. Fast to scaffold, shadcn components, decent visual output. AI _can_ drive this codebase – it's real React code, not generated output like Umso's export. The issue isn't AI-drivability. The issue is architecture: React Router, 60+ dependencies, client-side hydration, SEO penalties that matter, and the perpetual upgrade churn that comes with a dependency tree that size. The principle: **marketing sites should ship as static HTML by default**. React SPAs are the wrong tool for this job even when AI can work with them. **The sharper lesson from this attempt:** AI-drivability is necessary but not sufficient – the architecture still has to be right for the use case
  - **Astroship Pro + reference doc method (this session):** finally. The non-obvious thing that makes this work isn't Astro itself – it's the reference doc. AI can follow it. None of the previous approaches made AI collaboration clean
  - **Journey lesson:** the tool is less important than whether AI can actually drive it. GUI tools can't be driven by AI. Generated HTML can't be reverse-engineered cleanly. React SPAs can be driven but aren't the right architecture for a marketing site. A lean static template + a well-structured reference doc that describes every component in AI-readable terms = leverage. **The reference doc is the missing piece across all prior attempts** – and once that clicks, it retroactively explains why the others failed
- **Honest timing:** not literally an afternoon. Several focused hours of conversation with meaningful thinking between steps. "One session to scaffold, one reference doc, one commit to rebrand" is more accurate than "one afternoon". Don't oversell or the post loses authority
- **Core pattern – the reference-doc contract:** Split the work into two roles that can be agent-played in sequence. The reference doc (`docs/component-reference.md`) catalogues every component in the template with the detail a copywriter actually needs: props, slots, **content-shape constraints with specific word-count ranges** (hero headline max 10 words, testimonial quote 25–45 words, card description 15–25 words, FAQ answer 40–80 words), when-to-use vs when-not-to, visual notes, embedded screenshots of how each component renders. Plus design tokens (colour system, type scale, spacing rhythm), six section-level page patterns (landing / features / pricing / about / contact / interior), and a list of known gaps where the template doesn't ship what we need. It's not a component list – it's a contract-grade spec. (1) **Copywriter** – reads the reference doc, picks a section pattern, produces a per-page spec at `docs/specs/[page-name].md` that names specific components and supplies content fitted to their constraints. Copy is drop-in-ready because it was written against the constraints, not discovered at implementation time. (2) **Developer** – reads the spec + the reference doc, implements the page. The reference doc is the contract. Gaps are flagged as `CUSTOM: [component]` so the developer knows when to build rather than force-fit. **Distinctive move:** spending half a day on the reference doc up-front means every subsequent page build is close to one-shot-able. Most "AI codes my website" demos skip this and wonder why the output is generic. **The generalisation:** handoff fidelity is the first thing to collapse in agent workflows; a contract-grade reference doc fixes it for bounded problems – building out pages in a known template – which is narrow but concrete enough to learn from
- **The semantic colour alias pattern:** Astroship ships indigo-dominant. Renaming `indigo-*` classes to match a new brand colour in-place is the wrong move – it changes what the classes _mean_ without changing the visual. Better: define `brand` in `tailwind.config.cjs` as `colors.brand = colors.indigo`, then find-and-replace `indigo-` → `brand-` across every `.astro` file and rename the variant prop unions in Card/Pricing/Badge. Rebrand = one config edit. Code stays honest. Worth diagramming in the post

The next session can pick up cold and be productive immediately. Connects to the [[#The memory bank framework – signature concept, canonical reference post|memory bank framework]] idea earlier in this file – this is a worked example of pre-population at end of a scaffold session rather than initialising from an empty state

Check whether the callers you'd protect actually have content worth preserving before letting the additive-preservation rule auto-fire

Where (c) inherits a cached format convention without auditing it, (d) flattens an explicit user convention toward the common default. **Mechanism:** agents are trained on majority-voice writing, so non-dominant conventions read as errors to flag rather than craft to preserve. **All four shapes share the cumulative-failure quality** with the original Sharp-style instances: invisible until prompted, surviving across sessions, fix is a deliberate periodic pass not a one-time intervention. **Updated frame – sharper than the original dark corollary statement:** the failure isn't only about wrong-local defaults around _facts_; it's about any pattern where agent attention defaults toward the convenience of cached state and away from re-validation against fresh ground truth – plus one variant (d) where the default isn't cached state but the majority-voice prior the model was trained on, asserted against an explicit user preference. - **Verification loops for visual work – match the inner loop to the failure mode (architecture shipped, worked example pending):** The best-practices doc's verification examples (typecheck / test / lint) read as universal but implicitly assume a project whose failure mode is compile/runtime correctness – an API server, a library, a Node service. A marketing site's failure mode is visual: components look wrong on mobile, spacing drifts from spec, token inheritance silently breaks, a Tailwind class compiles but doesn't render because of a JIT cache miss, an icon resolves to an unexpected glyph, a slot renders empty because a component relaxation dropped a default. Build catches none of this. Phase 4 on fidero.com made the gap concrete: every Stitch-generated component needed eyes-on verification that `bun run build` couldn't provide, and the session accumulated a workaround dance documented in `active-context.md#Tooling` – dev-sandbox pages for icon-resolution fast-fails, `curl` + `grep '<svg data-icon>'` against the rendered HTML to confirm each icon landed as inline SVG, manual dev-server restarts when Tailwind's JIT stale-cached a new class on first use, explicit Phase 6 "eyes-on" checkpoints in `progress.md` for visual polish (marquee join-gap artefact, among others) the build would never see. None of these are bugs in the build step; they're evidence that the build step is the wrong layer for this failure mode, and the workaround stack is a Level 2 substitute for a verification tool that doesn't exist in the current loop. **The principle:** the verification loop should match the failure mode the project most often ships. Build catches compile-time correctness (types, imports, icon resolution at static prerender, Astro compile errors); browser inspection catches visual correctness (layout, spacing, responsive breakpoints, runtime JS, JIT-cached styles). The two are layered, not substitutes – dropping the build step would let type errors ship; dropping the browser layer is what lets visual drift ship.
