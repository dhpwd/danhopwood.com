- The memory bank framework – signature concept, canonical reference post. 6-file structured memory system for persistent AI context across sessions. Source: `~/cli-agents/shared/memory-bank.md`. Angle: "my AI forgets everything between sessions – here's the 6-file system that fixes it". Notes (Feb) – key questions and answers to structure the post around:
  - **"How do I initialise?"** → Prompt "initialise memory bank" with lean canvas or architecture docs attached for richer project-brief/product-context. Agent creates `memory-bank/` folder and all 6 files. For multi-repo: import a single shared file (`@~/my-instructions/memory-bank.md`) instead of copy-pasting
  - **"Meaningful difference vs compress?"** → Compress = lossy summary of a single session. Memory bank = structured, persistent, full-fidelity context across sessions. 5 separate files with update guidelines. Each new session loads all 5 → much richer and more predictable context for continuation than a compressed summary could ever produce
  - **"What about updating?"** → Instructions include an "update memory bank" trigger with steps. Run after each task. On trigger: review ALL files, even if some don't need updates
  - **Stale misleads harder than empty:** the memory bank's failure mode is quiet decay – a session ships work but skips the update, next session opens the files and starts from a confident-but-wrong orientation. Observed on Fidero: after the homepage spec landed, `active-context.md` still said the spec was the next deliverable and `progress.md`'s In progress section still said "awaiting homepage spec work". A fresh agent would read both with the authority of fresh context and waste time re-synchronising before realising the files were out of date. **Sharper framing:** stale context actively misleads – it reads with confidence and creates false orientation. Empty context makes the next session start fresh; stale context makes them start wrong. **Discipline:** memory bank updates belong in the same commit as the work they describe, or an immediate follow-up before session end. The framework's "only update when prompted" rule is about avoiding bloat, not deferring maintenance – prompt at session end, land the update, close out clean
  - **Bloat is the parallel failure mode:** staleness is one decay direction; bloat is the other. Observed on Fidero in the polish pass after Phases 1–3 landed: `progress.md`'s new "Done" bullets ran 5–7 lines each (narrative-leaning) while every older bullet ran 1–3 lines (summary-style); `active-context.md` grew an "Implementation notes" section that duplicated lessons already living in `system-patterns.md`; the same lessons appeared in three places. None of it factually wrong – just over-distributed. The shared `memory-bank.md` already has the rule (`Never duplicate full lessons in both`); the agent (me) violated it anyway and a deliberate polish pass caught it. **Fix:** extract lessons to `system-patterns.md` (their rightful home per the content-boundary rules), leave one-line pointers in `progress.md` and `active-context.md`, delete the duplication. Compression by relocation, not by information loss. **Discipline:** the "review all six files" step on memory bank update isn't just a staleness check, it's a distribution check – does each piece of information sit in its rightful home, or has it leaked into multiple files? The audit is bidirectional. **Mechanism:** agents producing volume default to assuming volume = thoroughness, so the bloat side of decay needs the same active counter-pressure as the staleness side
  - **Session workflow that connects it:** Never compact, never clear. Task done → commit → /exit → new session. /prime loads all memory bank files at start of each session. Compaction becomes unnecessary because sessions stay short and focused, and cross-session context lives in the memory bank not in compacted summaries
  - **The coherence argument:**
    - The obvious pitch is session continuity – "your AI forgets, here's how to fix it." But the stronger argument is architectural coherence
    - Without a bird's eye view, AI builds isolated solutions in-situ: duplicated logic, inconsistent patterns, responsibilities landing in the wrong place. It doesn't even register consistency as a concern – it's wired to complete the task in front of it, not to think about where the solution belongs in the wider system
    - Gets worse as the codebase grows. Each new task adds another isolated solution that doesn't know about the others
    - Two dimensions of code quality: (1) quality in individual pieces of code – AI is good at this, (2) coherence and consistency across the full app – AI is nearly incapable of this _by default_. Memory bank attacks dimension 2
    - The fix: memory bank gives the AI both the bird's eye view (product context, project brief) and the lower-level design conventions and patterns (system-patterns.md, tech-context.md) _before_ it touches anything. "Give it the blueprints first to orient, then give it the spec to execute against"
    - This reframes memory banks from "nice to have for context" to "architectural necessity" – especially for anyone past hobby-stage codebases
    - The continuity argument has a reasonable counter: "just use /compact." Nobody has a counter for the coherence argument
    - Find an analogy for "each piece is good, the whole doesn't hold together." The experience is universal for anyone whose AI codebase has grown – needs something that makes the mechanism click, not just a brand comparison

- **Rebuilding fidero.com with an AI agent – the reference-doc-driven workflow.** Scaffold + reference doc + memory bank + semantic colour alias for the Fidero marketing site rebuild, starting from the Astroship Pro v2.2.0 template. The repo itself is permanently private so the post IS the public artefact

The fourth is substantially less painful – not because Astro is magic but because the workflow finally lets an AI agent actually drive the build. The post's hook is the contrast between the three that didn't work and the one that does, and the non-obvious thing that makes the difference (a reference doc the AI can follow). Strapline candidates: _"Rebuilding fidero.com: I've done this four times. Here's what finally worked"_ / _"Three painful attempts and one reference doc"_ / _"Rebuilding fidero.com: one reference doc, two roles, one commit to rebrand"_ (first two are narrative, third is technical-spine)
  - **The journey – four attempts (hook material):**
    - **Umso (manual GUI):** evaluated alongside Webflow and Framer; Umso was the fastest of the three to get something live. "Fastest GUI" still means every piece of copy, every image, every section layout is constructed by hand. Scaling = manual labour per page. Death by a thousand clicks
    - **Astro port of the Umso export (older repo exists separately):** the intent was right – get off GUI, into code, make it AI-maintainable. The execution was agonising. Umso's exported HTML and JS is generated output, not hand-written code, and reverse-engineering it into a clean Astro project took an unreasonable amount of time. Lesson: you can't port a generated codebase into a maintainable one. Start clean
    - **Lovable (React SPA):** currently live at fidero.com. Fast to scaffold, shadcn components, decent visual output. AI _can_ drive this codebase – it's real React code, not generated output like Umso's export. The issue isn't AI-drivability. The issue is architecture: React Router, 60+ dependencies, client-side hydration, SEO penalties that matter, and the perpetual upgrade churn that comes with a dependency tree that size. The principle: **marketing sites should ship as static HTML by default**. React SPAs are the wrong tool for this job even when AI can work with them. **The sharper lesson from this attempt:** AI-drivability is necessary but not sufficient – the architecture still has to be right for the use case
    - **Astroship Pro + reference doc method (this session):** finally. The non-obvious thing that makes this work isn't Astro itself – it's the reference doc. AI can follow it. None of the previous approaches made AI collaboration clean
    - **Journey lesson:** the tool is less important than whether AI can actually drive it. GUI tools can't be driven by AI. Generated HTML can't be reverse-engineered cleanly. React SPAs can be driven but aren't the right architecture for a marketing site. A lean static template + a well-structured reference doc that describes every component in AI-readable terms = leverage. **The reference doc is the missing piece across all prior attempts** – and once that clicks, it retroactively explains why the others failed
  - **Honest timing:** not literally an afternoon. Several focused hours of conversation with meaningful thinking between steps. "One session to scaffold, one reference doc, one commit to rebrand" is more accurate than "one afternoon". Don't oversell or the post loses authority
  - **Core pattern – the reference-doc contract:** Split the work into two roles that can be agent-played in sequence. The reference doc (`docs/component-reference.md`) catalogues every component in the template with the detail a copywriter actually needs: props, slots, **content-shape constraints with specific word-count ranges** (hero headline max 10 words, testimonial quote 25–45 words, card description 15–25 words, FAQ answer 40–80 words), when-to-use vs when-not-to, visual notes, embedded screenshots of how each component renders. Plus design tokens (colour system, type scale, spacing rhythm), six section-level page patterns (landing / features / pricing / about / contact / interior), and a list of known gaps where the template doesn't ship what we need. It's not a component list – it's a contract-grade spec. (1) **Copywriter** – reads the reference doc, picks a section pattern, produces a per-page spec at `docs/specs/[page-name].md` that names specific components and supplies content fitted to their constraints. Copy is drop-in-ready because it was written against the constraints, not discovered at implementation time. (2) **Developer** – reads the spec + the reference doc, implements the page. The reference doc is the contract. Gaps are flagged as `CUSTOM: [component]` so the developer knows when to build rather than force-fit. **Distinctive move:** spending half a day on the reference doc up-front means every subsequent page build is close to one-shot-able. Most "AI codes my website" demos skip this and wonder why the output is generic. **The generalisation:** handoff fidelity is the first thing to collapse in agent workflows; a contract-grade reference doc fixes it for bounded problems – building out pages in a known template – which is narrow but concrete enough to learn from
  - **Reference docs are statements of truth, not logs (editing sub-lesson, part 1 – tone-scrubbing):** While capturing screenshots for the reference doc, the agent wrote the findings as "**Verified:** X is Y". Looked fine in isolation. I pushed back: _"shouldn't the document simply be a statement of the information needed vs a running history?"_ Right – the "Verified:" framing was process history masquerading as reference material, and every such note rots the moment it's written ("verified by whom? when? still true?"). The fix was to rewrite every instance as plain present-tense statements of current truth. **The principle:** reference docs state the present; logs state the past. Mixing them makes the reference doc decay because process notes become stale immediately. **Why this matters for AI-authored docs specifically:** agents naturally produce log-style output because they narrate what they just did ("I verified X", "I checked Y", "I found Z"). Steering them to present-tense statements of standing truth takes deliberate editing. Worth naming as a distinct editing pass when working with agents on reference material – AI-generated reference docs need a "strip the process history" sweep before they're actually usable as reference
  - **Reference docs need a second editing pass – usage-feedback (editing sub-lesson, part 2):** The homepage spec surfaced a second maintenance pass beyond tone-scrubbing. After using the reference doc on a real task (speccing the homepage section by section), the doc had visible blind spots – the scaffold pass couldn't see them because you don't know what you don't know until you've used the thing. Concretely, we went back and added: a new "Starting a page spec" section at the top of `component-reference.md` (scrape workflow, relax-before-new principle, pattern-naming rule, overrun-handling guidance), per-component "Relaxation candidate" notes inline against Hero/Logos/Features/FeatureGrid/Testimonials/CTA (so the next copywriter sees the lessons without re-discovering them), and an updated workflow summary at the bottom with `RELAXED`/`NEW` flag conventions, a two-table output format (Table A relaxations, Table B new components), and a cross-link to `homepage.md` as the first worked example. **The tighter frame:** reference docs need two distinct editing passes – **tone-scrubbing** (part 1) gets you a doc that's correct in isolation; **usage-feedback** (part 2) gets you one that's correct in practice. Scaffold pass produces the first; first real use of the doc produces the input for the second. Worth planning for as a deliberate checkpoint after any non-trivial task the doc is supposed to guide – otherwise the blind spots compound across every subsequent use
  - **Specs need their own review pass – strategic-consistency (editing sub-lesson, part 3):** Reference docs need tone-scrubbing and usage-feedback (parts 1–2 above); page specs need a third checkpoint – review against the memory bank before implementation starts. The homepage spec was scraped verbatim from the live fidero.com, which is fine as a content source, but the live site carries decisions that contradict Fidero's positioning strategy. Concretely: a "Manifesto" nav item that `product-context.md` had already flagged for replacement with an About page ("feels pretentious for Fidero's current stage"), and a "Solutions" dropdown linking to seven vertical and comparison pages whose survival into the rebuild is documented as TBD. Implementing the spec as-written would have shipped a homepage that contradicts strategy on one link and 404s on seven others. Review caught both before any code was touched. **The generalisation:** specs are inputs to implementation sessions, and once handed off they get executed without a human on every line – strategic consistency with the permanent project context (memory bank) is non-negotiable, otherwise the implementation session propagates errors that were invisible at drafting time. **Three editing checkpoints, three different questions:** tone-scrubbing asks _is this written as reference or as log?_ Usage-feedback asks _does this cover the blind spots first real use surfaced?_ Spec-vs-memory-bank asks _does this contradict anything we've already decided?_ Cumulative – each catches errors the others miss, and skipping any one means the post-hoc fix costs more than the pass would have
  - **Don't reproduce what the tool surface already provides (editing sub-lesson, part 4 – deduplication):** Parts 1–3 are about how to edit reference material. Part 4 is about _what belongs in it_ when the subject is an externally-defined thing with its own surface – a tool, an API, a schema. First draft of the memory bank's Stitch notes duplicated the tool schemas themselves: tool names, parameter descriptions, enum values, operational warnings – despite the next session loading all of that on demand via its own tool flow. Caught, trimmed. Second round still had meta-instruction redundancy ("Load schemas via ToolSearch before first use") – the harness already enforces this. Caught again, trimmed again. Third round finally converged on value-add only: **interpretation** (Stitch works in screens, not components), **project-specific values** (what to set colorMode/fonts/roundness/customColor to for _this_ template), **workflow integration** (Phases 4–5 only, unit-mismatch workaround), and **non-obvious watch-outs** (apply*design_system is a retrofit, not a generation hook; output format unknown until first run). **The principle:** reference material about an externally-defined thing should carry only what that thing doesn't describe itself. Everything else is accessible the moment the reader reaches for it. **Mechanism:** agents over-document because documenting feels like progress. Enumerating known facts is easy; omitting in favour of distilled interpretation is harder. The default is over-write, and the counter-pressure is to keep asking: \_does the reader already have access to this?* **Same structural beat as tone-scrubbing (part 1):** both are forms of over-writing reference material. Tone-scrubbing includes process history that belongs in logs; deduplication includes tool facts that belong in schemas. Same corrective question in both cases
  - **Cleanup pass as institutional editorial pressure (editing sub-lesson, part 5 – simplification):** Beats 1–3 (Sharp / relax / name-by-pattern) require live user push-back; the dark corollary observes that pattern capture alone doesn't prevent recurrence. The constructive answer is **structured self-review passes**, not user vigilance in the moment. Worked example: ran `/simplify` after the Phase 1+2 implementation diff (Marquee primitive extraction + five Astroship component relaxations), which spawns three parallel review agents (reuse / quality / efficiency) against the changed files. They collectively caught ten distinct architectural defaults the working agent had slipped through under "make it work" pressure – gradient-text inlined in three places (extracted to a shared `gradient-text.astro` primitive), CTA dark/light variants scattered across six ternaries (collapsed to a single `variants[background]` map), Hero `ctaLayout: "dual" | "single-with-trust-row"` mode flag coupling two orthogonal concerns (dropped in favour of inferring presence from `secondaryCta != null` and `trustItems?.length`), leaky `gapClass`/`paddingClass` string props on the Marquee primitive (removed), triple-nested logos rendering ternary (flattened), stale `{" "}` artefacts in feature markup, `<Fragment>` workarounds left over from earlier ternary shapes, missing `inert` on the marquee clone track, hand-rolled footnote normalisation, and stringly-typed mode flags. None were bugs; none affected the build. Left unaddressed, the codebase leaks architectural choices that should've been cleaned up, and the leakage compounds with every subsequent feature. **Pairs with beat 2 (relax-before-build):** beat 2 is editorial pressure at _spec time_ – pick the right approach before writing code; beat 5 is editorial pressure at _code-review time_ – catch the wrong-local defaults that slipped through anyway. Both necessary; neither sufficient on its own. Beat 2 catches things the spec author defaulted to wrong; beat 5 catches things the implementer defaulted to wrong despite the spec being right. **Sharper version:** the /simplify pass treats your own previous output as suspect – the same skepticism you'd apply to someone else's code. Agents producing volume default to assuming volume = thoroughness; structured self-review is the institutional counter-pressure. **Generalises:** any sufficiently large agent-driven implementation pass should be followed by a structured review pass against the diff, not just a build/test verification. Build passes confirm the code runs; review passes confirm the code is shaped right. Different question, different answer, different agent
  - **The semantic colour alias pattern:** Astroship ships indigo-dominant. Renaming `indigo-*` classes to match a new brand colour in-place is the wrong move – it changes what the classes _mean_ without changing the visual. Better: define `brand` in `tailwind.config.cjs` as `colors.brand = colors.indigo`, then find-and-replace `indigo-` → `brand-` across every `.astro` file and rename the variant prop unions in Card/Pricing/Badge.   - **The Sharp gotcha – agent-steering beat (1 of 3):** While writing `tech-context.md`, the agent described Sharp as an explicit dev dependency the Astroship template installs. I pushed back – image optimisation "just worked" on danhopwood.com and I didn't remember ever installing Sharp manually. The agent spawned a doc-researcher sub-agent against docs.astro.build, which came back confident: Sharp is Astro's default image service but "not bundled with Astro itself – listed as a peer/dev dependency that the project must provide." The sub-agent had misread the docs. **Lesson:** agents are persuasive at the confidence layer and soft at the ground-truth layer.   - **Relax before build – agent-steering beat (2 of 3):** First draft flagged 12 new components. User push-back: _"is it possible that some of them can be achieved by updating existing components? e.g. 'locked at 6' – we could update the component to make it more flexible, which feels better than creating a new component"_. Reassessed. Five of the 12 were really prop-level relaxations to existing components (Hero → named right-side slot + single-CTA-with-trust-row variant, Logos → variable count + optional marquee prop, Features → variable count + optional role-label + longer descriptions, Testimonials → variable count + `layout="marquee"|"grid"` prop, CTA → optional bullets + footnote + `background="dark"|"light"` variant). Each relaxation additive, stock behaviour kept as default, future pages inherit the flexibility for free. Count went from 12 new → 8 new + 5 relaxations. **Lesson:** agents default to "build parallel component" when "add a prop to the existing one" is cleaner and costs less architecturally. The signal is a spec with a suspiciously long list of customs – check whether each one is one prop change away from fitting. Same structural beat as the Sharp gotcha: plausible-but-suboptimal default, user intuition disagrees, re-check grounds the right answer. **Single-use exception surfaced during spec review:** the same draft proposed code edits to Navbar and Footer – drop the log-in link, remove the template wordmark, widen footer to 4 columns, replace hardcoded link arrays – without flagging them as Table A relaxations. Single-use components skip Table A entirely and get treated as straight content replacement – an explicit escape hatch to the relax-before-build default when the stretching itself costs more than it saves
  - **Name by pattern, not instance – agent-steering beat (3 of 3):** Immediately after the relax-vs-build push-back: _"make sure new components are designed so they're reusable. 'tabbed-example-viewer' sounds specific to viewing examples versus being a tabbed component for any situation"_. Correct – first draft had `tabbed-example-viewer`, `topology-diagram-3col`, `declarative-vs-imperative-card`, `step-timeline-3col`, all named after the specific homepage content that motivated them. Renamed to `tab-panel`, `flow-diagram-3col`, `comparison-card`, `step-timeline`. Pattern is generalisable; each homepage section is an instance. **Mechanism:** agents name things after the task they just completed, not the reusable pattern. They're wired to complete what's in front of them, not to hoist the abstraction. Steer explicitly
  - **Three wrong-local defaults, one editorial pressure (meta-frame across beats 1–3):** Sharp = trust the doc over intuition (wrong-local on a factual claim). Relax-before-build = build parallel instead of modifying existing (wrong-local on architectural choice). Name-by-instance = name after the thing in front of you, not the reusable pattern (wrong-local on abstraction level). Three different failure categories, same editorial counter-pressure. Might be the post's spine: agents default to locally-optimal, user steers to globally-optimal, and the non-obvious thing is that you have to steer on architecture and abstraction – not just on facts

The dark corollary is **cumulative – every session adds evidence**, and the audit-style instruction (`use astro mcp to confirm…`) is no defence either: structured verification triggers can themselves become the surface where the wrong-local defaults reassert
