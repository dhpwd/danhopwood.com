## Setup

- **Angle (KPOI):** This is the fourth attempt at fidero.com. The first three were painful in different ways. The fourth is substantially less painful – not because Astro is magic but because the workflow finally lets an AI agent actually drive the build. The hook is the contrast between the three that didn't work and the one that does, and the non-obvious thing that makes the difference: a reference doc the AI can follow. Strapline candidates: _"Rebuilding fidero.com: I've done this four times. Here's what finally worked"_ / _"Three painful attempts and one reference doc"_ / _"One reference doc, two roles, one commit to rebrand"_ (first two narrative, third technical-spine)
- **The journey – four attempts (hook material):**
  - **Umso (manual GUI):** evaluated alongside Webflow and Framer, and the fastest of the three to get something live. But 'fastest GUI' still means every piece of copy, every image, every section is built by hand. Scaling = manual labour per page. Death by a thousand clicks
  - **Astro port of the Umso export:** the intent was right – get off the GUI, into code, make it AI-maintainable. The execution was agonising. Umso's exported HTML and JS is generated output, not hand-written code, and reverse-engineering it into a clean Astro project took an unreasonable amount of time. Lesson: you can't port a generated codebase into a maintainable one. Start clean
  - **Lovable (React SPA):** currently live at fidero.com. Fast to scaffold, shadcn components, decent visual output. AI _can_ drive this codebase – it's real React, not generated output. The issue isn't AI-drivability, it's architecture: React Router, 60+ dependencies, client-side hydration, real SEO penalties, and perpetual upgrade churn from a dependency tree that size. The principle: **marketing sites should render as static HTML by default**. The sharper lesson: AI-drivability is necessary but not sufficient – the architecture still has to be right for the use case
  - **Astroship Pro + reference doc method (this session):** finally – a lean static template the AI can actually drive. The difference-maker isn't Astro, it's the reference doc (covered below)
  - **Journey lesson:** the tool matters less than whether AI can actually drive it. GUI tools can't be driven. Generated HTML can't be reverse-engineered cleanly. React SPAs can be driven but aren't the right architecture for a marketing site. A lean static template + a well-structured reference doc that describes every component in AI-readable terms = leverage. **The reference doc is the missing piece across all prior attempts** – once that clicks, it retroactively explains why the others failed
- **Honest timing:** not literally an afternoon. Several focused hours of conversation with meaningful thinking between steps. "One session to scaffold, one reference doc, one commit to rebrand" is more accurate than "one afternoon" – don't oversell or the post loses authority

## Core patterns

- **The reference-doc contract:** Split the work into two roles played by an agent in sequence. The reference doc (`docs/component-reference.md`) catalogues every component in the template with the detail a copywriter actually needs: props, slots, **content-shape constraints with word-count ranges** (hero headline max 10 words, testimonial quote 25–45, card description 15–25, FAQ answer 40–80), when-to-use vs when-not-to, visual notes, embedded screenshots of how each renders. Plus design tokens (colour, type scale, spacing), six section-level page patterns (landing / features / pricing / about / contact / interior), and a list of known gaps. It's a contract-grade spec, not a component list
  - **(1) Copywriter** reads the reference doc, picks a section pattern, produces a per-page spec at `docs/specs/[page-name].md` naming specific components with content fitted to their constraints. Copy is drop-in-ready because it was written against the constraints, not discovered at implementation time
  - **(2) Developer** reads the spec + the reference doc and implements. Gaps are flagged `CUSTOM: [component]` so the developer knows when to build rather than force-fit
  - **Distinctive move:** spending half a day on the reference doc up-front makes every subsequent page build close to one-shot-able. Most 'AI codes my website' demos skip this and wonder why the output is generic. **The generalisation:** handoff fidelity is the first thing to collapse in agent workflows, and a contract-grade reference doc fixes it for bounded problems (building pages in a known template) – narrow, but concrete enough to learn from. **The payoff is front-loaded – it shows on the _next_ page, not this one:** the homepage was the expensive build that _created_ the component vocabulary; the product page that followed, a full platform deep-dive, needed _zero_ new components – every section mapped to one already built for the homepage, the opener the only one to earn fresh _relaxations_ (a text-only Hero variant and a mid-phrase-accent slot), additive prop-extensions rather than new builds. The reference-doc half-day amortises across every page after the first. **And the vocabulary outlives its motivating instance – the contract turns dead code into inventory:** a tabbed example viewer and its detail card were built for the homepage, cut from it in a spec rework before launch, and sat with zero live call sites – kept only as documented vocabulary, their reference entries noting the retirement. A later page's spec composed with them sight unseen (the copywriter works from the contract, never the repo), and they came back as that page's centrepiece: a 12-section platform deep-dive that needed exactly one new component. Without the contract, unused components are deletion candidates; under it they're stocked inventory, because what keeps a component alive is its entry, not its call sites
- **The contract unlocks pipelining – and the one shared surface left is the git index.** The two roles don't only run in sequence; because the contract keeps their inputs disjoint (the copywriter needs the vault + reference doc, the developer the spec + repo), they can run _concurrently on different pages_ – the next page's spec authored in one session while another session builds the current page. **In steady state the handoff channel is the working tree itself:** the spec session leaves its revisions as uncommitted diffs, and the build session treats the diff as the work order ("update the page from the spec diff"), pairing spec + implementation in one commit so the committed spec never asserts a state the site doesn't have – the working tree is the inbox, the commit is the merge. The diff is the work order only while the revision is _incremental_: a re-authored spec makes the deltas the wrong read – they anchor the build to what changed rather than to the new structure – so a rewrite hands over the whole document, and "read it in full, not the diff" is part of the work order. The channel even works mid-flight: a spec revision landing while the page is being built surfaces as a file-change notification, and the build adopts it in place rather than shipping the superseded line. This sits beside, not against, the captured rule that parallel _builds_ don't pay: two developers collide on the shared component layer and mint duplicate components, but a copywriter and a developer collide on almost nothing. Almost – the repo itself is the residual shared surface, and the collision fired: a habitual `git add -A` on a review-fix commit swept the other session's untracked spec draft into a commit it had no business in. The standing rule ("only commit changes made in this conversation – other sessions may be working concurrently") sat in always-loaded context and didn't outpull the convenient command – the dark corollary on a one-flag habit – and the catch came from the commit _output_, not the rule: an unexpected `create mode` line and ~300 surplus insertions in the stat. **Tells, recovery, and the real fix:** the fingerprint of a foreign change riding along is any unexpected `create mode`/`delete mode` line or surplus churn in the commit stat – and the sweep takes deletions as readily as additions (a concurrent session's file *removals* commit as if authored; recovery is mechanical once seen – restore the paths from HEAD and amend). Stage by explicit path and read the stat rather than trusting what the commit was meant to contain. But the durable fix is process, not another agent rule: one session per worktree removes the shared surface instead of guarding it. A staging rule drafted into CLAUDE.md was deliberately reverted on exactly this ground – codifying a guard against a state the process shouldn't produce is rule-bloat, the instruction-artefact version of defending an unreachable path – so when a failure needs two parties to collide, fix the cheaper party (session hygiene) rather than arming the other against it
- **The worked example is load-bearing, and it drifts (the method's hidden tax).** The method leans on a canonical worked example – the homepage spec – that the agent models every new page spec on, so when the implementation moves on but the example doesn't, the example silently teaches the obsolete pattern and every spec derived from it inherits the drift. Concretely: a polish pass moved section eyebrows to a muted default in the shipped code and deleted the `badgeColor` pass-through, but the homepage spec still wrote `color="brand"` on all ten eyebrows – so the product-page agent, drafting from that example, copied `color="brand"` onto all seven of its own, a value the component can no longer honour, and it would have shipped contradicting the live homepage. **Two compounding causes, both needed in the fix:** the example had drifted (reconcile it to shipped reality), _and_ the reference doc's `Sectionhead` entry never stated that its `badge` slot drops the colour prop (close the gap, so the ambiguity that made the example look plausible is gone) – the under-specified contract is the accomplice, because the example can only mislead in the space the contract left undefined. **Principle:** in an example-driven agent workflow the canonical example is production input, not documentation – it carries the same staleness liability as code and propagates to everything modelled on it, so it needs the same lockstep discipline. **Tell:** a derived spec that applies some attribute _uniformly_ (an accent on every eyebrow) is a fingerprint of example-copying – uniformity is rarely authored, it's inherited. **Generalises** to any concrete example baked into an always-loaded artefact: the build-page skill named `index.astro` as the page to model, where "the closest shipped page" is the durable form – a named example pins to a moving target, so state the rule and let the agent resolve the instance
- **The spec is a hypothesis until a build tests it – a rule that collides with a legitimate case earns a bounded carve-out, not a forced component.** The contract mostly propagates spec → code, but the traffic runs both ways: first contact with a real implementation is where an over-reaching rule shows itself, and the disciplined response is to refine the rule, not bend the component to it nor break it silently. Building the §4 illustrations against the image-style spec (§6.4) falsified two of its rules – a mono micro-label register with no legibility floor (fine in the abstract, too small to read at real size) and a flat "never a background wash" absolute that one focal-hub card legitimately wanted to break (a faint brand ground reading as 'this is the central node'). Both were fixed in the spec: a stated floor, and a **bounded, named exception** scoped like the spec's existing ones – the carve-out sits beside the established hero-glow exception, faint-ground-on-the-focal-node-only, behind white content – so the break is deliberate, a later review can't flag it as a defect, and a later session can't 'correct' it back to the rule. The bound is the whole game: an unscoped exception erodes the rule, a scoped one documents its real edge. **The trigger was a comparison, not a spec read** – holding the spec-era components against the older pre-spec ones surfaced both directions at once: what to propagate up into the spec (the legibility floor, the focal-anchor convention the stronger new work had found) and where the spec had over-reached – and the call was the user's, explicitly licensed (_"the image style isn't set in stone … that's cause to refine our image style, not force existing components to match"_). **And the contract can live in another repo:** the image-style spec sits in the central brand vault, not the site repo, so the revision is a cross-repo lockstep – the code change in the site's PR, the spec change a separate commit in the vault – and the off-repo half is the one trivially left behind. **Principle:** a spec authored top-down is a hypothesis, not settled law; treat a rule that fights a legitimate case as a candidate for an exception codified and scoped next to it, and remember the contract's reach (and its lockstep) can exceed the code repo
- **Claims-integrity guard rails: encode what the copy may say in the spec, where reviewers can police it – and codification transfers the check, not the verdict.** AI-authored marketing copy drifts toward the stronger claim, and the drift reads fluent: a proof callout rendered a customer's published "increased the amount of conversion data sent to Google by about 65%" as "put around 65% more conversion data through to Google Ads… the signals Google's bidding had been missing" – a narrowing (Google → Google Ads, algorithm → bidding) that sounds equivalent and isn't what was said. It was catchable only because the spec carries the rule next to the claim ("must stay a paraphrase of the published quote, never drift into claims the quote doesn't support"), so a spec-adherence reviewer flagged it unprompted – no human re-read the source quote. The triage is the second half of the lesson: the finding is neither agent-fixable nor agent-refutable, because whether "Google Ads" is faithful to a marketer discussing ad performance is a judgement about meaning and acceptable risk – the codified rule turns the agent into a tripwire while the verdict stays human, the boundary case of codify-the-triage: some rules transfer detection without transferring the call. **Generalises:** every externally-anchored claim (a customer quote, a benchmark, a compliance statement) wants its anchor _and its allowed envelope_ stated in the spec, because the copy will be reworked by sessions that never saw the source
- **The semantic token alias pattern:** Astroship ships indigo-dominant. Renaming `indigo-*` classes in-place is the wrong move – it changes what the classes _mean_ without changing the visual. Better: define `brand` in `tailwind.config.cjs` as `colors.brand = colors.indigo`, then find-and-replace `indigo-` → `brand-` across every `.astro` file and rename the variant prop unions in Card/Pricing/Badge. Rebrand = one config edit. Code stays honest. **The move generalises beyond colour (N=3):** the same mechanic – centralise a design-scale dimension as a named semantic token so re-tuning the whole register is one config edit with zero component touches – recurred with a **radius token layer** (`rounded-panel` 12px / `rounded-control` 8px, replacing scattered `rounded-2xl`/`rounded-lg`/bare-`rounded`) and the **heading type baseline** (Geist weight + tracking + line-height via an `addBase` plugin, not per-component `font-bold`). Colour, radius, type baseline – any design dimension that should hold a coherent register across the site wants to be a token, not a literal repeated at call sites. **Anti-pattern that makes the token earn its name:** don't redefine a _stock_ scale value (e.g. override Tailwind's `borderRadius.2xl` to mean 12px) – the class name then lies about its value, the exact trap the alias avoids by introducing a _new_ name (`brand`, `rounded-panel`) instead of mutating an existing one. **The rebrand's blind spot is every colour role that isn't 'brand':** a rebrand scoped to the brand alias leaves the feedback and accent roles on stock values – a Bootstrap-red error border and a stock red→orange "Popular" gradient pill survived the full rebrand and several polish passes, invisible until the user felt them as "jarring, off-brand, likely stock" (right on all counts) – so colour deliberateness means a decision per named role (error, success, highlight), not per brand hue. The fix added `danger`/`success` aliases with the same one-edit re-tune property, and half the softening was _application_, not hue: a 400-stop border where stock used 500 – lighter touch, not just cooler ramp. **And the boundary: not everything semantic should be a token.** The dashboard reds/ambers/greens inside the product illustrations stay literal by design – their job is imitating _other tools'_ status language, and routing them through the site's `danger` alias would re-theme a quotation
- **Peer-bucket mining via firecrawl (the rebrand-direction beat):** The alias is the _mechanism_ for swapping a brand. The harder question is what to swap _to_, and the default mistake is reasoning from aesthetic instinct. Better: name 4–6 peers whose visual register your prospects would mentally bucket you alongside, and pull their real design tokens via `firecrawl_scrape` with `formats: ["branding"]` – font families, colour roles, type sizes, shadow values, component metrics. Cross-reference the set: what's _shared_ is the structural signature, what's _individual_ is each one's differentiator. For fidero.com I pulled Stripe, Linear, Vercel, Resend, Mintlify and Neon as the 'premium infra' bucket. The shared signature was overwhelmingly typography, neutrals and accent restraint – not hue. **The non-obvious payoff:** Astroship's stock Tailwind indigo was already _inside_ the Stripe/Linear hue range. 'Looks templated' wasn't a colour problem. The lift came from Geist + composed heading weight + tight tracking + flat accent + tighter button radius – none of it a colour decision. **Principle:** reasoning from first principles over-weights the most-visible attribute (hue) and under-weights the harder-to-see ones (tracking, weight, restraint). The data shifts the weighting. **Generalises** to any design call where "what's the bucket convention" matters more than "what's individually best" – form layouts, button shapes, error-state UX. **Pairs with the alias beat:** alias handles the _swap_, peer-mining handles the _target_
- **Memory bank + CLAUDE.md handoff:** End of session, we wrote a 26-line CLAUDE.md (imports the shared memory-bank instructions) and pre-populated all six memory bank files with real content from the scaffold decisions, not stubs. The next session picks up cold and is productive immediately. Connects to [the memory bank framework](/posts/the-memory-bank-framework) – a worked example of pre-population at the end of a scaffold session rather than initialising from empty

## Agent-steering beats

- **The Sharp gotcha (beat 1 of 3):** While writing `tech-context.md`, the agent described Sharp as an explicit dev dependency the Astroship template installs. I pushed back – image optimisation 'just worked' on danhopwood.com and I never installed Sharp manually. The agent spawned a doc-researcher sub-agent against docs.astro.build, which came back confident: Sharp is the default image service but "not bundled – listed as a peer/dev dependency the project must provide." The main agent relayed that with confidence. I pushed back again. The agent then checked `node_modules/astro/package.json` directly – Sharp sits in `optionalDependencies`, auto-installing transitively unless the install fails. The sub-agent had misread the docs. **Lesson:** agents are persuasive at the confidence layer and soft at the ground-truth layer. When a claim is one `grep` away from certainty and the user's intuition disagrees, ground-truth-check before relaying
- **Relax before build (beat 2 of 3):** First draft flagged 12 new components. Push-back: _"is it possible some can be achieved by updating existing components? e.g. 'locked at 6' – we could make the component more flexible, which feels better than creating a new one"_. Five of the 12 were really prop-level relaxations to existing components (Hero → named right-side slot + single-CTA-with-trust-row variant, Logos → variable count + optional marquee, Features → variable count + optional role-label, Testimonials → `layout="marquee"|"grid"`, CTA → optional bullets + footnote + dark/light). Each relaxation is additive: stock behaviour stays the default, and future pages inherit the flexibility for free. Count went 12 new → 8 new + 5 relaxations. **Lesson:** agents default to "build a parallel component" when "add a prop to the existing one" is cleaner and cheaper. The signal is a spec with a suspiciously long list of customs – check whether each is one prop change from fitting
  - **Single-use exception:** the same draft proposed edits to Navbar and Footer without flagging them as relaxations. _"the navbar is one component where flexibility isn't really important as it's only ever used once"_. **Relax when there's a second caller, edit in place when there isn't.** Relaxing single-use components adds prop surface with zero reuse payoff. A refinement on the same axis: 'preserve stock defaults' only holds when the existing callers have opinions worth defending – when they're placeholder demo content awaiting rewrite (the other Astroship pages), defending them is defending the template against its own replacement, so flip the default globally
  - **Three editorial moments, not two:** the same default fires at three points, each more expensive to catch than the last. **Spec time** (beat 2 – pick the right approach before code), **wire-up time** (mid-assembly, when a genuine gap appears against the spec – pause and extend the relaxed primitive rather than inline-hacking the consuming page), and **review time** (the `/simplify` pass, Part 5 – catch what slipped through). Phase 6's homepage assembly surfaced three real relaxations mid-wire-up (Sectionhead `as?` + `badgeColor?` + optional desc slot, Logos text-tile fallback, Review initials-circle fallback) caught only because the session instruction explicitly said _pause and extend the primitive, don't inline-hack_. Without that counter-pressure, the locally-optimal inline workaround wins under 'make it work'
- **Default the canonical, mark the exception:** Beat 2 handles _whether_ to extend a component. This handles _what default to pick_ once extended. The canonical case should be the default. The exception carries the annotation. `GradientText` gained `variant?: "gradient" | "flat"` – first instinct kept the existing `gradient` as default, which would have meant 12 call sites annotated `flat` and the 'stock' default with zero callers. Inverting to `flat` collapsed all 12 to bare `<GradientText>`. Same for `Badge` (`"chip" | "eyebrow"`): _"does the same principle apply to badge? not a leading question"_ – re-flipped so `eyebrow` is default and `<Badge variant="chip">` is the one exception (at `card.astro`), which now reads literally as "I want a chip _instead of_ the canonical eyebrow." **Principle:** count the calls. N-canonical-to-1-exception → default canonical, mark the exception. 1-canonical-to-N-exceptions → you probably have _two components in a trench coat_ and should split them
- **Name by pattern, not instance (beat 3 of 3):** _"make sure new components are reusable. 'tabbed-example-viewer' sounds specific to viewing examples versus being a tabbed component for any situation"_. First draft had `tabbed-example-viewer`, `topology-diagram-3col`, `declarative-vs-imperative-card`, `step-timeline-3col` – all named after the homepage content that motivated them. Renamed to `tab-panel`, `flow-diagram-3col`, `comparison-card`, `step-timeline`. **Mechanism:** agents name things after the task they just completed, not the reusable pattern. They're wired to complete what's in front of them, not to hoist the abstraction. Steer explicitly
- **Three wrong-local defaults, one editorial pressure (the spine):** Sharp = trust the doc over intuition (wrong-local on a fact). Relax-before-build = build parallel instead of modifying existing (wrong-local on architecture). Name-by-instance = name after the thing in front of you (wrong-local on abstraction). Three failure categories, same counter-pressure. The non-obvious thing: you have to steer on architecture and abstraction, not just on facts
- **`!important` is a smell for a cascade shortcut – audit the bangs:** A polish session accumulated six Tailwind `!important` modifiers. User heuristic, stated directly: _"whenever I see bangs (!) in CSS it makes me think it's hacky – review and confirm best practices. not a leading question"_. The audit found **5 of 6 avoidable**: three `!max-w-none` were redundant (the element already sat in a `max-w-2xl` parent, fighting a constraint that wasn't binding); `!leading-[1.12]` had a clean responsive form (`lg:leading-[1.12]` wins at the same breakpoint, no bang); `!mt-10` was only needed because the element was a `space-y-5` child whose sibling selector out-specifies a child `mt-*` – the fix was to lift it out of the `space-y` flow. One bang survived (a genuine caller-override of a component default). **The sharp bit: the agent had _introduced_ several of those bangs earlier in the same session as the convenient fix, and they all _worked_** – a `/simplify`-style review wouldn't have flagged them (correct given the approach). Only the user's code-smell heuristic caught them. **Principle:** in agent-generated CSS a bang reliably marks where the cascade was shortcut rather than solved – is this overriding something not actually binding (redundant), something a breakpoint-matched utility would win cleanly (responsive), or something I'm fighting because the element is in the wrong place (restructure)? **Generalises** to any cheap, always-wins framework escape hatch – `!important`, `any`/`as` casts, `eslint-disable`, `# type: ignore` – all high-signal places to audit, because the cost structure pushes the agent toward them under volume
- **Cross-surface consistency is a blind spot the agent won't cover unprompted.** The integration logo bar carried an arbitrary order inherited from the original scrape, and it survived _every_ prior session that actively edited that component (adding logos, re-sizing them, fixing the wrap) – the agent treated the given order as settled until a user _"sanity-check the order"_ surfaced it. Sharper still: the page named GA4 and BigQuery in the copy (GA4 was even a column in the hero illustration) while the logo bar omitted both – the agent had authored or handled _both_ surfaces and never cross-checked them; the user caught the gap, not the agent. **The resolution principle:** the agent's first re-order optimised the obvious local metric (recognition – lead with the biggest names); the sharper frame, supplied by the user, was to order by _problem-association_ – lead with the platforms the visitor connects to the felt pain – so the bar reinforces the page's thesis rather than just listing vendors. **Two failures, one shape:** the agent doesn't re-audit decisions that arrive as givens, and doesn't notice when two surfaces it controls contradict – both because re-checking inherited state reads as wasted motion against a 'just make the change asked for' frame. **Generalises** to any build with multiple authored surfaces that must agree (copy vs visuals, nav vs page inventory, schema vs docs) and any inherited decision riding along unexamined because no one re-opened it. **The coupling includes rationale, not just content:** a layout decision whose stated reason depends on a content property ("lead sits at the top so the column has enough height to balance the card") is silently invalidated by any copy edit that changes the property – a spec-side copy trim deleted the sentence carrying that height, and the dead space surfaced two passes later, caught by the user. A rationale that references length, count or order is a consistency edge between copy and layout, and a copy pass doesn't know the edge exists. **Peer of Part 3 (strategic-consistency):** that catches spec-vs-permanent-context drift; this is the content-vs-content, asset-vs-copy sibling, and takes the same corrective – a deliberate consistency pass, not in-the-moment vigilance.
- **Reframe an embedded-solution request to the register – don't execute the suggestion literally.** An open ask often carries its own proposed fix – _"is there anything we can do to make the homepage more delightful e.g. using animation?"_ – and the wrong-local default is to build the named thing: bolt on the genre-standard scroll-reveal entrance animations. That default is wrong on two axes at once. **Register:** for a precise/infra-grade voice the on-brand form of delight is motion that _demonstrates the product_ – the hero illustration enacting the silent-failure thesis as it scrolls in, the 'engine' node pulsing, data flowing the pipeline connectors – and the genre-default decorative reveal is the move to _exclude_, lowest register-fit and the one every templated site already makes. **Correctness:** the naive scroll-reveal also ships a hidden-content footgun – start-state baked into the markup (`opacity-0`, a zeroed counter) means no-JS visitors, `prefers-reduced-motion` users and search crawlers see the blank state permanently. The fix inverts it: the static _end_-state is the authored markup, and JS applies the armed state only when motion is allowed, so the animation is pure enhancement over correct content. The sharper move still was arguing partly _against acting_ – the page was already through its polish passes, and the real remaining gap was content (an unfilled proof slot, an image-weight pass), not motion. **Principle:** a request shaped 'improve X, e.g. do Y' embeds Y as a candidate, not a spec – the real question is 'what does improve-X mean for this register', with 'do something else' and 'do nothing' kept on the table. The agent default treats the suggestion as the instruction. The move treats it as one option and reframes to the goal.
- **Control anchoring by sequencing the inputs – withhold prior art until the fresh draft exists, then mine it.** A page with earlier incarnations (fidero.com has been built four times) invites reading the old versions 'for reference' first – which anchors the agent to their structure and framing before it has formed its own, so it recreates rather than authors. The user withheld them on purpose: _"I deliberately didn't get you to read the old pages up front as I didn't want to skew your work."_ The corrective is a sequencing rule, now baked into the workflow: draft fresh from the canon, **then** open the old versions as a post-draft harvest. **Harvest discipline:** lift individual lines that still earn their place, adapt each to current voice and positioning, take the structure or framing from neither – old assets carry stale conventions (pre-current pronoun discipline, unqualified metrics) and sometimes an internal-only framework miscast as a public spine (an internal 5-layer sales diagnostic had been the backbone of an old product page – sales scaffolding, never public structure). The product page netted two adapted lines out of two old versions otherwise left whole. **Principle:** anchoring is cheaper to prevent than to undo – govern _what the agent sees and when_, don't only re-validate after. **Prevention sibling to the dark corollary:** it re-validates cached state after the fact, while this stops the cache from forming. **Pairs with Part 3:** inherited content (live-site scrape or old-repo version) carries decisions that contradict current strategy – same corrective, here sequenced after the fresh draft so it can't anchor
  - **Sibling gate – ask for what the canon can't supply.** Before drafting, surface the decisions strategy doesn't settle (which pricing tiers are public, which verticals a page covers) one question at a time until confident. The shape of the gate follows from derivability: **ask for what the agent can't derive, propose what it can.** A spec page asks, because page scope isn't in any doc. The blog/share skill inverts it – its content basis carries the angle, so it proposes a plan and waits for a redirect rather than interrogating

## Dark corollary and companions

- **Dark corollary – pattern capture doesn't prevent recurrence.** Documenting a pattern is no defence against re-entering it under volume. The default still pulls in the moment and needs active counter-pressure. The sharper frame: the failure isn't only wrong-local defaults around _facts_ – it's any pattern where agent attention defaults toward cached state (or the majority-voice training prior) and away from re-validation against fresh ground truth. Cached state feels authoritative, fresh validation feels like wasted motion, and the agent takes the cheaper read – including from one source that hides in plain sight: the agent's _own just-completed change_, which silently lifts or adds a constraint while the agent keeps reasoning from the pre-change model. After a refactor makes a table fit its container, pinning a sticky header the old overflow box had made impossible stays un-considered, because "overflow traps sticky" is still the cached constraint though the refactor just removed it – "what did my change just enable or break?" is the re-validation the agent omits by default. Five instances of the externally-cached version:
  - **Relay-confidence** (the Sharp spine) – trust an upstream summary instead of ground-truth-checking. Recurs even inside work the user flagged as careful: an image-optimisation audit misread `awk` output to claim `sharp` sat in `dependencies` (it was `devDependencies`), then framed the `^0.33.2` pin as "holding sharp back" when Node's closest-wins resolution means Astro uses its own nested copy regardless. Both caught only on re-check
  - **Source-vs-summary substitution** – when the user names a specific doc as the check target, read the memory bank's summary instead of the source. _"Did you read those docs?"_ The forced second pass through `component-reference.md` surfaced two findings the summary had hidden
  - **List-write-without-resort** – append to a backlog in think-of-it order without re-checking dependencies or file placement. A migration item sat below the upgrade that named it a prerequisite. Six items duplicated across `active-context.md` and `progress.md`. None caught until _"are the backlog items in the correct order?"_
  - **Convention-inheritance without audience check** – continue a cached format convention without testing it against the viewing environment. `component-reference.md` used Obsidian `[[wikilinks]]` (75 of them) that nothing in the repo renders, inherited transitively from an earlier session
  - **Micro-convention dismissal** – flatten an explicit user convention toward the majority default. The use–mention quote rule (single for word-as-word, double for literal strings) got framed as 'not the dominant standard'. It's actually well-established (Quine), just uncodified by newspaper guides. Agents trained on majority-voice writing read non-dominant conventions as errors to flag rather than craft to preserve
- All five are invisible until prompted, survive across sessions (cumulative – every session adds evidence), and take the same corrective: structured periodic re-validation against the canonical source (file, list state, framework docs, audience fit, voice rules), driven by deliberate editorial passes rather than user vigilance in the moment. Same structural fix too – capture each override in its canonical home (`product-context.md#Voice rules` for voice, `system-patterns.md` for code) so it's inherited, not re-litigated
- **Agent memory is a second canonical home, and the agent lets it drift unattended (companion).** The structural fix above turns on _one_ canonical home per fact, re-read each run so it's inherited not re-litigated. Giving a review sub-agent its own persistent memory (`memory: project` on the spec- and convention-checkers) quietly breaks that: the agent wrote a parallel convention store into the repo that _restated_ rules already authoritative in the curated docs it re-reads every orientation – reproducing, autonomously, the same-fact-in-two-places drift the discipline exists to prevent. And it's invisible – the frontmatter flag auto-enables Write/Edit (against the agents' read-only-review contract) and lands the files in version control as a side effect, no edit you authored. **The fix is to withhold, not to curate:** where a project already has a strong human-curated single source of truth the agent re-reads each run, agent memory is redundant by construction and a drift engine, so the canonical docs stay the only home. Persistent agent memory earns its place only where knowledge must accumulate across runs _and isn't otherwise captured_. **Principle:** knowing when to _withhold_ an agent capability is as load-bearing as when to grant it, and a capability with autonomous-write side effects (memory, self-editing config, auto-commit) reproduces the very failure modes the human discipline guards against, minus the human in the loop to catch them. **Generalises** to any agent handed a writable cache that shadows an authoritative source – convenient until it's a competing truth
- **Steering tags hard-code a corrective for a specific agent reflex (companion).** The base failure is reflexive-agreement: when _challenged_, the agent reaffirms its prior position rather than re-deriving. The user defeats it with an inline tag – _"not a leading question"_ (and family: _"play devil's advocate"_, _"steelman the other side"_, _"what am I missing"_) – mandating fresh reconsideration. Helpful reads as agreement, so the reflex is to confirm when asked "are you sure?". Treat the tag as a reconsider-mandate and re-derive from scratch – a walkback is the editorial output the question asked for, not face-loss. **The same tag has a second job, fired _before_ a position exists: licensing honest evaluation against the show-me-your-work sycophancy reflex.** _"Audit the prominence of the brand colour… your honest judgement… not a leading request"_ pre-empts the validate-what-I'm-shown default. The tell it worked is a _calibrated_ read – the audit came back "most of this is in good shape, the one real issue is the accent on every heading", not manufactured fault-finding. An honest assessment is allowed to be mostly-positive – the license removes the thumb from the scale, it doesn't mandate criticism. **A third job, fired _after_ a review is delivered: forcing iteration past the agent's too-lenient first pass.** _"re-check your assessment with fresh eyes. Proceed with everything worth doing. Flag any changes of mind first."_ A single review pass calibrates lenient – the agent treats its just-delivered assessment as settled, re-examination reading as wasted motion (the dark corollary turned on its own output). The tag re-runs the pass, and the tell it worked is _reversals_, not just new finds: across iterations the agent dropped a change it had argued for, downgraded a 'gap' to 'leave', and re-cut its own fresh wording. Changes of mind are the licensed output, the way a walkback is what 'are you sure?' asks for. **A fourth job, fired when the challenge is relayed from a second agent: neutralising deference to the peer.** Relaying another AI's input – _"another AI suggested this, they might be wrong"_, _"do you think these are an improvement? not a leading question"_ – adds a deference pull the bare user-challenge lacks: the relayed position reads as considered work from a capable system, so the agent can rubber-stamp it the way it rubber-stamps a sub-agent's summary. The flag licenses a merits-read, split by claim-type. A peer's _factual_ claim is a ground-truth case – verify, don't relay (the peer's "the opener CTA won't render large" was one grep from false – the shared Link primitive already defaulted large). A peer's _argument_ is a re-derive case – concede what lands and hold what doesn't (a peer's case for a route rename reframed the call around an IA pairing the agent hadn't weighed, and the honest output was a partial update, not capitulation). The tell is the same calibration the other jobs want: neither rubber-stamp nor reflexive defence. **The unifying read:** these phrases are bug reports about agent reflexes – confirm-on-challenge, validate-on-show, settle-on-first-pass, defer-to-a-relayed-peer – with the corrective compiled into the wording, the user's phrasing doing the agent's self-regulation for it
- **False-equivalence to close a decision (companion – and _not_ a cached-state instance).** The dark corollary turns on a stale read preferred over fresh validation; this is the opposite footing – the facts were fresh and correct, and the failure was _editorial_: flattening their significance into a false equivalence to land a tidy recommendation. "Cal.com's built-in payment and a standalone Stripe Checkout come out about the same – the VAT invoice should come from the accounting system either way" folded two materially different options into one and quietly foreclosed a call that was the user's to make. The underlying facts were right (Cal.com charges via a raw Stripe `PaymentIntent`); what got understated, because it complicated the clean recommendation, was that the PaymentIntent records _zero tax_, shows the customer no VAT breakdown, and issues no compliant invoice. The user named the move: _"this made it sound like the two are effectively equivalent."_ **The corrective was a concrete artefact, not an argument** – a screenshot of the real Stripe checkout (postcode → VAT line → gross total) the abstraction couldn't talk around, with a doc-researcher then confirming at source that `PaymentIntentCreateParams` carries no tax fields. **Mechanism:** once converged on a recommendation, disconfirming detail reads as friction and gets minimised – "these are effectively the same / it doesn't really matter" is the highest-risk shape because it _removes_ a decision rather than informing it, and it's most seductive once the agent has already picked a side. **Tell:** an equivalence or "doesn't matter much" arriving _alongside_ a recommendation the agent already favours – the equivalence is doing work for the conclusion. **Corrective:** when the call is the user's, lay out the real differences at full weight _before_ recommending, and treat a user's concrete artefact (screenshot, real receipt, measured value) as ground truth that outranks the agent's abstract case. **Inverts the claims-integrity beat** – there AI copy drifts toward the _stronger_ claim; here agent judgement drifts toward the _flatter_ one – same root: the cheaper cognitive move (converge, simplify) over the costlier one (hold the distinction and weigh it)
- **Stitch production runs – the spec discipline transferred (positive companion):** After the dark corollary had captured 'guessing Stitch tool schemas' and 'guessing theme values' as recurring failures, Stitch got real production runs – four clean single-attempt generations (`silent-failure-illustration`, `flow-diagram-3col`, `comparison-card`, `tab-panel`) off the same disciplined spec → prompt handoff. The interesting thing is _why_: not that the agent stopped defaulting wrong, but that the structural correctives caught up – (1) a Stitch Tooling section in `active-context.md` with template-baseline values and 'what's not obvious from the schemas' notes, loaded automatically by `/prime`; (2) the copywriter spec already existed at section-level granularity, so the prompt was assembled by lifting exact strings and visual intent rather than improvising; (3) the prompt treated as a downstream of the spec, just one more consumer of the contract. **The generalisable pattern:** the spec discipline scales beyond copywriter→developer to copywriter→Stitch→developer (and any other generation tool). Stitch is just another reader of the contract. **Connects to the dark corollary:** capture pays off only when the captures get embedded structurally (memory bank notes the next session loads, verification baked into the workflow, spec discipline that scales to new consumers) rather than relying on the writer to remember
  - **Tooling watch-out – the quality lever isn't the default.** The MCP's `generate_screen_from_text` defaults `modelId` to unspecified → Flash. The UI also defaults to Flash. The good model (`GEMINI_3_1_PRO` / "Thinking with 3.1 Pro") only gets used if you go looking. First-time callers silently get the worse model. Good MCP usage isn't 'use the tool', it's 'audit the schema for quality levers the wrapper doesn't surface'. One residual translation step too: Stitch's `designMd` steers visual vocabulary but not implementation choices like icon library – output used Material Symbols despite the designMd asking for fluent, so a Material Symbols → astro-icon swap is now a fixed step in the memory bank
  - **One informative failure – the hand-build fallback.** `step-timeline` timed out MCP-side and didn't land server-side either. The constructive move: hand-build it from the spec using the already-generated `flow-diagram-3col` connector as the template – identical quality, faster than retrying. **Heuristic:** if a similar pattern has already been through the workflow, one failed Stitch attempt is enough signal to fall back to hand-building against the prior run's translation pattern. If not, wait out the server-side finish. **Deeper point:** structural embedding of Stitch lessons compounds into hand-build capability – once the connector was captured as a reusable primitive in `system-patterns.md`, the fallback was ~30 minutes, not from-scratch. The workflow's value isn't only "Stitch works when you structure the prompt", it's "once you've built one instance of a pattern you're no longer dependent on Stitch for the next"

## Verification

- **Match the inner loop to the failure mode – "a build pass is not a render pass":** The best-practices doc's verification examples (typecheck / test / lint) implicitly assume a project whose failure mode is compile/runtime correctness – an API, a library, a service. A marketing site's failure mode is _visual_: components wrong on mobile, spacing drifting from spec, token inheritance silently breaking, a Tailwind class that compiles but doesn't render (JIT cache miss), an icon resolving to the wrong glyph, a slot rendering empty because a relaxation dropped a default. Build catches none of it. **The architecture is three layers, each catching what the one below can't:**
  1. `bun run build` – the mandatory fast-fail. Cheapest feedback: type errors, broken imports, astro-icon resolution at static prerender
  2. **Chrome MCP** – the runtime layer for any UI change. `read_page` for DOM, screenshot / `gif_creator` for layout, `read_console_messages` for runtime and hydration errors
  3. `curl` + grep – demoted to fallback for cheap binary 'is this HTML string present' checks
  - **Placement is the substantive move:** the terse directive lives in `CLAUDE.md#Before completing any task` (read every session), the detailed pattern + anti-patterns in `system-patterns.md#Component verification workflow`, and `active-context.md` keeps a one-line pointer. The eventual upgrade per 'promote persistent rules into code': a `Stop` hook that blocks completion until `bun run build` passes and optionally checks Chrome MCP was used on any UI-touching session – turns the checklist from a suggestion into a gate
- **Worked example – the bug class the beat was waiting for (`tab-panel`):** The click handler toggled the HTML `hidden` attribute on panel bodies that also had the Tailwind `flex` class. `bun run build` passed clean. `curl` + grep passed clean – the `hidden` attribute was literally present on the three non-active panels. Both layers greenlit code that was actively broken: Tailwind's `flex` wins the cascade over HTML `hidden` (user-agent stylesheet is the lowest specificity tier), so all four panels rendered at once. The **only** layer that caught it was the Chrome MCP screenshot after the first test click – one visual comparison and the bug was obvious. Fix: toggle the Tailwind `hidden`/`flex` classes via `classList` instead of the HTML attribute. A second bug from the same component: the script used a plain `<script>` with no `astro:page-load` wrapper, so it would silently die after any ClientRouter navigation (bundled module scripts run once per visit). `/simplify` caught it. The docs MCP confirmed the fix _and_ revealed that the existing `contactform.astro` `is:inline` + `{ once: true }` precedent is defensive-for-inline-scripts, not the canon to inherit for bundled ones. **The third verification layer, then: a docs check against whatever framework contract the new code leans on** – especially when the basis is a codebase precedent rather than first principles, because precedents ossify under a framework that's moved on
- **The aesthetic layer is the human eye, not the agent's.** What the three layers actually encode is _who's judging_. Layers 1 and 3 are agent-judgeable (pass/fail, match/no-match). Layer 2 is agent-_capturable_ but not agent-_judgeable_ – `read_page` and `gif_creator` produce artefacts, but "does this _look_ right" requires the user. In a polish session the agent took ~10 captures and built two A/B compare pages. The user's eye was ground truth on every aesthetic call (diagnosed weight when it was font, missed a typeface-metric quirk, overshot global tracking). **The flip-toggle compare primitive** does the work for sub-perception deltas: one container, two stacked images, `Space`/arrow to swap in place – side-by-side fails because the eye normalises each independently and deltas like warm-paper `#FAFAF8` vs pure white vanish. ~80 lines of throwaway HTML, worth naming as a reusable pattern. **The agent's job in layer 2 is to surface artefacts cheap enough that the human will look** – there is no MCP for "does this feel right", and there shouldn't be. **The artefact's scale must match the property being judged.** A local treatment is judgeable from a single component render – but a _systemic_ property (over-use, rhythm, repetition across the page) is invisible at component scale: you can't see "the accent is on every heading" from one heading. The brand-colour prominence call was settled by rendering the whole page twice – current vs proposed, one variable flipped – and judging on the real scroll. That's the exact inverse of the isolate-in-a-sandbox default – sound for a component's _intrinsic_ correctness, but blind to systemic aesthetics _and_ to interaction correctness. A z-index collision between a component's sticky header and the page's global sticky navbar (both `z-20`, the navbar being the page's stacking ceiling) is invisible at component scale and at rest; it surfaces only in the assembled page at the scroll state where the two overlap, and verification that tracks the changed element's _own_ behaviour sails past it unless that collision state is deliberately driven. Anything composing with a global sticky/fixed/stacked element carries the same scale demand as a systemic aesthetic: judge it in the full page, at the state where it actually collides. **And the inspection zoom distorts size the other way:** a magnified screenshot – the agent's own tool for reading detail – over-represents legibility, so a size-dependent call (is this type too small) passes from the zoom and fails at 100%. Micro-labels that read fine in every verification zoom were too small on a real laptop screen, caught by the user's eye, not the agent's magnified captures. A size property is judged at the real viewport scale, never from a zoom
- **On a taste call, generate divergent candidates – including your own – not a single recommendation.** Layer 2 has the agent surface artefacts and the eye judge. The sharper move for a decision with _no spec-correct answer_ (which treatment, which direction) is to _widen_ the option space, not narrow it to one option and ask for a thumbs-up. The testimonial restyle was settled by a 6-way rendered bake-off in a throwaway sandbox – baseline, two hand-built treatments, and three Stitch `GEMINI_3_1_PRO` generations (clean / editorial / results-forward) against the synced design system, every candidate rendered in the _real_ §10 grid with the _real_ copy so each was judged in situ. **Two things make it more than 'render some options':** the agent's own hand-built designs compete on equal footing with the tool's, so the call is 'which direction' not 'accept or reject Stitch', and the agent has proposals in the ring rather than only a verdict on the tool's output; and the candidates are _deliberately divergent_ – different devices, not A/B variants of one – so the bake-off explores the _direction_ space, where the flip-toggle compare primitive resolves sub-perception deltas _within_ a chosen direction. **The agent default to steer against:** converge to a single recommendation early (I opened with a lean). And descriptions don't substitute for renders on taste – ASCII mockups in the question prompt didn't settle it and the user escalated to real candidates – so the move is to spread the set and let rendered output carry the decision. **Stitch's role flips:** divergence engine, not the spec's executor – same tool, opposite job. **The output can be a recombination, not a pick:** a form-field bake-off resolved as a blend – the Stitch generation's input surface with a hand-built candidate's label weight, mono placeholders kept only where the hint is a literal token – because rendered candidates decompose into orthogonal decisions (surface, labels, placeholders) that recombine across entries. And the tool can contribute the winning _idea_ rather than a winning design: its surface treatment extended the site's own card grammar to form controls – the payoff of keeping the generation tool's design-system asset synced, so it generates inside the project's grammar instead of adjacent to it. **For layout calls there's a zero-build tier below the sandbox:** toggling classes on the live DOM in the controlled browser renders each candidate for a screenshot without writing a line – cheap enough that the user's proposed variant gets rendered _before_ the agent argues, which is how a rendered user idea beat the agent's argued recommendation on the same question
- **But measurement is the agent-judgeable visual layer – and it ground-truths past _the user's_ plausible-but-wrong cause.** A large subset of seemingly-visual problems reduce to a measurement, and there the agent is ground truth and should measure rather than eyeball or defer. The browser MCP exposes the tools: `getComputedStyle` (type metrics, spacing), `getBoundingClientRect` (geometry, gaps, overlap), `canvas` + `getImageData` (luminance). The throughline across a polish session: _the user reliably flags the symptom and often offers a cause, and the offered cause is frequently wrong_ – e.g. _"the line-height looks cramped"_ → computed styles showed both headings identical at ratio `1.000` (both too tight, because `text-5xl`/`text-6xl` bake `line-height:1`); _"some logos are hidden"_ → canvas luminance proved one asset was a light mark rendering near-white, not a layout bug; _"the spacing seems related to the callout"_ → measurement found a pre-existing `mt-20` unrelated to the callout work. This is the **Sharp gotcha mirror-imaged**: there the wrong cached state was the agent's and the user's intuition was right; here the wrong cached state is the user's and measurement is right. Same discipline, opposite source. **Corollary – verify the effect, not the edit:** an `mt-8`→`mt-12` bump reported as done was a visual no-op (the `space-y` parent out-specified it), caught with _"looks the same as before"_ – a CSS edit needs an _effect_ check (measure the rendered value), not just a code-diff check. **The discipline extends off the page:** a scraped Customer.io mark arrived as the vendor's pale dark-mode variant (near-invisible on a white card), and the right green wasn't recalled, it was measured – pulled as `--color-verdant-600` (oklch) from customer.io's own production CSS and converted to hex. Brand facts the eye can't check are where a confident training prior ships a plausible wrong answer that _survives review_, because no reviewer knows the hex either – an external value gets ground-truthed at its source like any other number. **Boundary:** the eye stays ground truth for aesthetic judgement (warmth, balance, whether a weight reads chunky); measurement is ground truth for anything that reduces to a number (contrast, leading, gaps, alignment). Reach for measurement first, escalate to the eye for the genuinely-aesthetic residue
- **The verification loop is itself cached state.** Scaffolded in Phase 1 for compile-time failure modes (brand find-and-replace, template reorg, memory-bank scaffold), it wasn't re-validated as the work shifted to visual component generation. Tool-availability drift is the specific trigger: the Chrome MCP tools were available for several sessions before the guidance was updated to use them, leaving a latent contradiction between `CLAUDE.md`'s "load in browser" rule and the Stitch block's "curl the sandbox" workaround. **Discipline:** when the dominant work type changes (compile-heavy → visual-heavy, backend → frontend, single-pass → iterative polish) _or_ when new tools land in the harness, re-validate the verification loop rather than inheriting it from the scaffold session. Same cached-state failure mode as the dark corollary, one layer up – what's re-validated is the tooling _fit_, not the tooling accuracy
- **A skill's terminal workflow is an inherited global default too, and reversibility is the gate (companion).** The beat above re-validates the verification _loop_ when the project shifts. The same cached-global-default shape sits in a skill's _hand-off_ steps: `/ship` hard-coded its endpoint as `push → open a PR`, simply wrong for a project whose declared convention is a local `--no-ff` merge, user-initiated, no PR – and the agent followed the skill toward a PR until the user stopped it, because an imperative step in a loaded skill out-pulls a project fact it never thought to reconcile. **The fix parameterises the endpoint on project-declared policy, and the axis deciding automate-vs-hand-off is reversibility:** run the cheap reversible steps (a PR is one command and deletable), but stop at the first irreversible or outward action the project gates (a merge into the default branch) and hand over the exact command. **Principle:** no global default gets a terminal action right for every project, so the irreversible step defers to project policy rather than firing – the same gate the harness already puts on outward and destructive actions belongs inside any skill that ends in one. **Distinct from Part 10:** that scrubs a skill's _prose_, this is its _workflow shape_ – an always-loaded instruction artefact carries a content liability _and_ a behaviour liability, and project-fit is the behaviour one
- **A skill assumes its own applicability – check the hard precondition against the stack before paying its cost (companion).** A loaded skill frames everything as 'how to do this here', never 'can this run here at all', so its load-bearing precondition goes unchecked until the work is already underway. The Claude Design lane (`/design-sync`) is a multi-step, multi-agent skill whose whole premise is rendering the project's components in a web canvas – and it renders _React_: it needs a compiled component bundle the design agent mounts at runtime. This site is Astro – components compile to static HTML with no runtime form, and `dist/` is rendered pages, not a component bundle – so the lane is inapplicable by construction, not by configuration. The catch was cheap and early: read the repo shape (no React in the dependency tree, `dist/` is static pages) against the tool's actual contract before running the skill, rather than discovering it several agents deep. **Principle:** an agent skill's framing creates momentum toward execution; its hardest assumption – the stack or runtime it presupposes – is the first thing to falsify, because it's seconds to check and hours to hit late. **Generalises** to any tool whose value rests on a stack premise: a component-design tool needing a component runtime, a codemod needing a parser for your dialect, a profiler needing a build mode you don't ship
- **The verifier has its own failure modes – rule out the tool before debugging the code.** Iterating on a logo band (rapid rebuilds + hard reloads + in-page `fetch`/`img.src` churn) saturated the Astro dev server's on-demand image pipeline (Sharp, serial): the browser rendered _blank_ logos and avatars, which reads identically to a broken build. The agent's own failure was over-persistence – it re-drove the same browser path (navigate, scroll, wait, screenshot) ~10 times, and even reached for a throwaway static-file server, before asking "is the tool broken, or the code?". The disambiguation is cheap and independent: `curl` one image URL (a fast 200 means the server is healthy and it's browser-side contention) and `bun run build` (pre-generates every image, so a clean build means the assets are fine). **The non-obvious escape:** layout is measurable while the render is blank – Astro emits `width`/`height` on every `<Image>`, so `getBoundingClientRect` reports correct geometry (wrap, row-count, gaps) before a pixel paints. The question actually asked – "is the logo row still one line?" – was answerable with zero images loaded, which made the static-server workaround pure wasted motion. **The same dev server lies in more than one way:** its Tailwind JIT cache is a second stateful subsystem with the identical trap – introduce a utility class that's new to the whole codebase (an arbitrary value like `leading-[0.6]`, a first `text-5xl`) and the long-running server serves the class in the DOM with no matching rule in the bundle, so the element renders unstyled while the code is correct; a `bun dev` restart re-runs the JIT scan and fixes it. The naive verifier-check has its own false-negative here – grepping the dev-served `base.css` finds nothing, because JIT utilities live in a Vite virtual module rather than that file, so the right oracle is the production build's emitted CSS or the rendered computed style. **And the browser harness has the same shape, one tool out from the dev server:** the Chrome-MCP-controlled tab runs _backgrounded_ (`document.visibilityState === "hidden"`), and browsers throttle `IntersectionObserver`, `requestAnimationFrame` and CSS animations on hidden documents – so a scroll-triggered or looping animation never fires under automation, and the element sits frozen in its pre-animation state, which reads identically to a dead observer or a script that never ran (fully-visible element, clean console, a freshly-attached observer that also won't fire). The independent oracle is to stop waiting on the paint: foreground the tab (a single screenshot does it, and the motion then plays through) or drive the trigger directly – clear the play-once flag, dispatch the `astro:page-load` event, assert the element re-arms – so you verify the mechanism, not the throttled render. CSS _transitions_ freeze the same way with a sneakier read-out: `getComputedStyle` on a transitioning property returns the _from_-value indefinitely, while every layer of the cascade checks out – rule present in the stylesheet, selector matching, even an injected `!important` ID rule reading back wrong – and `:focus` doesn't match at all while the document lacks focus, though `document.activeElement` is set. The diagnostic is one property before any cascade theory: `document.visibilityState` / `document.hasFocus()`. **Principle:** when verification comes back red, rule out the verifier first – a stateful tool (a dev server with a cache or an on-demand pipeline, a browser tab that's backgrounded) has failure modes that masquerade as the code's, and an independent oracle (curl, build, geometry-from-attributes, a forced foreground) separates them in seconds. **A capability gap is the inverse of a lie** – the harness can't _reach_ the state under test rather than misreporting it. The controlled browser couldn't size its viewport below the `lg` breakpoint, so the responsive layout's mobile branch – _the_ marketing-site failure mode, components wrong on mobile – was the one condition it couldn't produce; verifying the reachable branch and assuming the rest is the trap. The first move is to reach the state through a narrower context rather than simulate it: inject a same-origin fixed-width `<iframe src="/page" style="width:390px">` via the JS tool, and media queries, `matchMedia` and viewport units all resolve against the _iframe's_ viewport – the page genuinely renders its mobile branch, scrollable and assertable through `contentWindow`, a real 390px run rather than a proxy for one. Synthesis is the fallback for triggers no nested context can supply – force the box's `<lg` overflow mode and read the collapsed `grid-template-columns` off `getComputedStyle`, exercising the branch without the condition – and that's the move that generalises to `prefers-reduced-motion`, RTL, a feature flag. Reach the state where a context can carry it; synthesise only the residue. **Distinct from "the verification loop is itself cached state":** that beat re-validates the loop's _fit_ when the work type changes; this one says the loop's _tools_ can lie in the moment, and the corrective is a second, cheaper oracle.

## Editing sub-lessons

These are the editing passes AI-authored reference material needs – reference docs, memory bank files and skills alike. Each asks a different question. They're cumulative – none is sufficient alone.

- **Part 1 – tone-scrubbing (is this written as reference or as log?):** Capturing screenshots for the reference doc, the agent wrote findings as "**Verified:** X is Y". _"shouldn't the document simply be a statement of the information needed vs a running history?"_ The "Verified:" framing is process history masquerading as reference – it rots the moment it's written ("verified by whom? when? still true?"). **Principle:** reference docs state the present, logs state the past. Mixing them makes the reference decay. Agents naturally produce log-style output because they narrate what they just did ("I verified X", "I checked Y") – steering to present-tense standing truth takes a deliberate pass. The same principle extends temporally to `progress.md`: a bullet written mid-iteration (`tracking-widest 0.12em`) goes stale when the value changes twice more before the session ends – draft iterative-work bullets at the _end_, describing the landed state. Path is recoverable from `git log`. Standing state is what the next session needs. And a log's _facts_ outlast its _verdicts_: "we chose contained horizontal scroll" stays true forever, but "contained scroll is the correct fit" turns into a lie the moment a later session reworks the layout – so a changelog records what changed, not what's right, because a future reader can't tell a still-true verdict from an expired one. Drafting at session-end fixes stale _values_ but not this – the verdict was final when written, and falsified afterward. And verify the landed state from the diff, not from session memory: a progress bullet recorded a grid change the commit never contained, and the false record then corroborated the equally-stale spec against the actual page – the changelog's claims need the same ground-truthing as any other cached state
- **Part 2 – usage-feedback (does this cover the blind spots first real use surfaced?):** After using the reference doc on a real task (speccing the homepage section by section), the doc had visible blind spots the scaffold pass couldn't see – you don't know what you don't know until you've used the thing. We added a "Starting a page spec" section (scrape workflow, relax-before-new, pattern-naming, overrun handling), per-component "Relaxation candidate" notes inline, and a workflow summary with `RELAXED`/`NEW` flag conventions and a two-table output format. **The frame:** tone-scrubbing gets a doc correct in isolation. Usage-feedback gets it correct in practice. Plan it as a deliberate checkpoint after the first real task the doc guides
- **Part 3 – strategic-consistency (does this contradict anything we've already decided?):** Page specs need a review against the memory bank before implementation. The homepage spec was scraped verbatim from live fidero.com, which carries decisions that contradict positioning strategy – a "Manifesto" nav item `product-context.md` had flagged for replacement, and a "Solutions" dropdown linking to seven pages whose survival is documented TBD. Implementing as-written would have published a homepage that contradicts strategy on one link and 404s on seven. **The generalisation:** specs get executed without a human on every line, so strategic consistency with permanent project context is non-negotiable – otherwise the implementation session propagates errors invisible at drafting time
- **Part 4 – deduplication (does the reader already have access to this?):** When the subject is an externally-defined thing with its own surface (a tool, an API, a schema), reference material should carry only what that thing doesn't describe itself. First draft of the Stitch notes duplicated the tool schemas (names, params, enum values) the next session loads on demand anyway. Second round still had meta-instructions the harness enforces. Both trimmed. Third round converged on value-add only: **interpretation** (Stitch works in screens, not components), **project-specific values** (what to set for _this_ template), **workflow integration**, **non-obvious watch-outs**. **Mechanism:** agents over-document because documenting feels like progress – enumerating known facts is easy, distilling to interpretation is harder
- **Part 5 – structured cleanup via `/simplify` (is the code built right, not just running?):** Beats 1–3 require live user push-back. The dark corollary says capture alone doesn't prevent recurrence. The constructive answer is structured self-review passes, not user vigilance in the moment. Running `/simplify` after an implementation diff spawns three parallel review agents (reuse / quality / efficiency) against the changed files. It reliably finds wrong-local defaults the implementer slipped through under 'make it work' pressure – none bugs, none affecting the build, all leakage that compounds with every subsequent feature. **The sharp version:** `/simplify` treats your own previous output as suspect, the same scepticism you'd apply to someone else's code. Build passes confirm the code runs. Review passes confirm it's built right
  - **The slip kinds are characteristic of the _phase_, not the writer** – any single pass only sees its own phase's defaults. Across runs they spanned: within-file architectural style (inlined gradient-text extracted to a shared primitive; dark/light variants collapsed from six scattered ternaries to a `variants[background]` map; a Hero `ctaLayout` mode flag dropped in favour of inferring presence from `secondaryCta != null` and `trustItems?.length`; leaky `gapClass`/`paddingClass` string props removed from the Marquee primitive; a triple-nested logos ternary flattened; missing `inert` restored on the marquee clone track; hand-rolled footnote normalisation removed; dead `<Fragment>` / `{" "}` scaffolding from earlier ternary patterns cleaned up); cross-file duplication (a chevron-pill connector byte-identical in two components, extracted to `flow-connector.astro`); framework-lifecycle correctness (the `tab-panel` script bug above); TypeScript surface drift (a wrapper hand-redeclaring a primitive's enum instead of importing `BadgeColor` – export the type from the primitive, import it everywhere); layout-baked-into-primitive (`prose-with-callout` baking a `Container` wrapper a grid-column consumer couldn't use – parameterise the wrapper as opt-in via `constrained?`); comment-narration bulk (~25 lines of design-rationale comments violating "no comments unless WHY", one already factually stale within the same diff – canonical home is `system-patterns.md`, not the component file); deploy-cleanup blockers (~1.8 MB of throwaway A/B scaffolding in `public/` that Astro would have served as crawlable production assets – the efficiency reviewer's byte-footprint check is the layer that catches it)
  - **What `/simplify` misses:** it catches slips _within_ a chosen default, not when the default itself is inverted. The `Badge` API-default inversion surfaced from the user (_"does the same principle apply? not a leading question"_), not the reviewers – they checked the API for internal consistency given the chosen default, not whether the default was right. The user remains the editorial layer for "is the API itself right?"
  - **`/simplify` is also a doc-editing pass.** A run surfaced a finding in `docs/component-reference.md` too – when primitives were extracted and `prose-with-callout` gained `constrained`, the reference doc needed new entries to stay in lockstep. The reference doc is the contract. Every contract-changing edit should touch it in the same commit. **And lockstep runs sideways, not just forward:** a new component's entry was written to justify its own choices (muted dot bullets, argued against brand checks) while the sibling CheckBullets entry's anti-pattern banned exactly the re-inlined marker row the component shipped – two locally-correct entries holding contradictory law, each fine in isolation, caught only by a conventions pass that read both. The author defends the decision it just made against the objection it had in mind, not against the standing rules its addition touches. Writing the entry for what you added isn't enough: reconcile the neighbouring entries whose claims now collide, or amend their scope in the same pass – here the fix was a variant on the existing primitive, dissolving the contradiction in code rather than wording. **Quantified claims are the contradiction magnets:** a reference-doc absolute ("the _only_ sanctioned use of an arbitrary text size") is falsified silently by every sibling that ships one, because the new code never touches the claiming file – three components carried the 'forbidden' pattern before any pass noticed the sentence. Enumerate, don't quantify: name the sanctioned instances, so adding one forces the edit and the claim degrades into a stale list (visible) rather than a false law (invisible)
  - **The review pass has become a stack, and a recall-biased `--fix` needs triage.** `/simplify` is one axis (reuse / quality / efficiency on the changed code). A page build now front-loads a _structural_ pass before the tactical one: `/structural-review` runs a spec-adherence check and a convention-conformance check in parallel – does the implementation match what the spec said, and how the codebase already does things? – and only then does `/code-review` run the multi-angle bug-and-cleanup finders. Structural-before-tactical is the same ordering as the editing parts: catch 'this contradicts the spec' before 'this line has a bug'. **The sharper lesson is downstream, on _applying_ a high-recall review.** `/code-review` at high recall is built to over-surface – it errs toward flagging – so its findings are candidates, not a worklist, and `--fix`-ing them wholesale is its own wrong-local default. The pricing-page run returned three kinds and only the first is a fix: a real in-scope defect (an `aria-label` on a role-less `<svg>`/`<span>`, which screen readers silently drop without a companion `role="img"` – the attribute was present in the built HTML and inert in the a11y tree, so a grep confirming it 'landed' is a false pass); a real finding whose fix lands _outside_ the reviewed diff (a logo array duplicated into the homepage – correct to extract, wrong to do from inside a pricing-page review that would then quietly edit a page nobody asked it to touch); and a flagged 'inconsistency' that's an _intentional_ divergence the finder can't recognise as one (a tier rendered brand-violet on the cards but orange in the comparison table – the orange is the table's stand-in for the cards' orange 'popular' border, so 'fixing' it would delete a decision). A fourth kind is the _correct-but-premature_ extraction: a recall reviewer flags a real duplication the instant it appears twice, but minting the shared abstraction on the second instance tends to produce a near-duplicate with a divergent wrapper, so the extraction earns its place only at the third consumer – a true duplication finding can still be a _defer_, on maturity grounds rather than scope. **Principle:** a recall-biased reviewer's whole value is coverage, which makes its output pre-triage by construction – the application judgment (fix now / defer as out-of-scope / reject as intended) doesn't transfer to the tool, and `--fix` is most dangerous on the findings it's _most right about_, the ones that are correct but shouldn't be applied here. **Except: the judgment transfers exactly as far as it's been codified.** Once the premature-extraction call was captured as a memory-bank rule ('don't extract on two divergent instances – wait for the third consumer'), a verification pass handed that rule refuted a real-duplication finding with no user in the loop – the triage call moved from user judgement to agent-performable check. Each triage decision you write into the project's canonical docs is one the review stack can make next time; the uncodified residue is what still needs the human. **Two more finding-classes for the taxonomy, both on the remedy side.** The suggested fix can itself be the bug: a simplification finder proposed collapsing `bullets && bullets.length > 0` to `bullets?.length && …`, which in JSX-style templates renders a literal `0` for an empty array – the original 'noise' was the guard. A finder's patch carries its own defect rate and needs the same verification as its finding. And the known-issue shield: a captured global deferral (the backlogged image-pipeline migration) pattern-matched a finding to 'already tracked' and nearly suppressed a one-line local fix that banked most of the win immediately (~220KB → ~45KB of emitted webp) – 'defer as known' is a triage bucket that's wrong whenever a cheap partial fix exists ahead of the structural one. **Generalises** to any auto-fixing linter or codemod run at recall: the scope boundary, the intentional exception, and the defective remedy are the classes it reliably gets wrong, and all three read as plain bugs from inside the diff. **And the contract itself can be the defect: the stale-spec finding.** A whole-artefact spec-check (the full page against the full spec, not a diff review, after both have moved) returned its must-fixes _against the contract_ – the spec demanded a treatment a documented polish pass had deliberately moved past, and a layout value that had never shipped at all. The trap in the second: the spec _and_ the progress log both recorded the value as landed – written records corroborating each other – while `git log -S` showed the commit landed the opposite. Deliberate-or-drift is an archaeology question and the commit is the only arbiter; "source of truth" status doesn't exempt the spec from being the side that's wrong
- **Part 6 – framework-drift check (is this still true against the framework's current docs?):** Any reference material wrapping a third-party framework needs periodic re-validation against that framework's docs, because the framework moves under your feet. User-prompted check against the Astro docs MCP found six real findings in <15 minutes: `src/content/config.ts` riding an Astro 5 backcompat path that disappears in 6.0, with blog hero images typed `z.string()` and bypassing `astro:assets` (the `image()` schema helper is the fix, unoptimised until then); `<ViewTransitions />` renamed `<ClientRouter />` in 5.0; the blog `[slug].astro` using a legacy entry API the v5 Content Layer removes; Astro Actions as the framework-native ContactForm path; an `image.layout:'constrained'` entry missing its required pair `image.responsiveStyles: true` (without both you get the responsive markup but no resizing styles – the pit of failure); and the marquee's sanctioned-but-flagged-'advanced' `Astro.slots.render() + set:html` pattern carrying no documented caveats, so the constraints captured in `system-patterns.md` are inferred, not sourced. **Principle:** the framework's own docs are the source of truth, not the cached project summary that was correct once. **Tool gate:** these were near-free because the docs were an MCP – the economic claim is that drift checks should be done more often _because_ a good MCP collapses the cost from minutes-per-finding to seconds. **The check runs forward, not only as periodic re-validation – verify an external API at the moment you author a claim about it into the contract.** A spec or reference doc that names third-party surface (an `astro-seo` `noindex`/`nofollow` prop, an `@astrojs/sitemap` `filter` signature) makes a factual claim about an external contract from _recalled_ knowledge, and recalled API is itself unverified cached state – a spec asserting a wrong API is worse than a chat message that does, because the build gets written against it. Same cheap MCP doc-check, moved to authoring time as a gate (_"before we do anything else, verify the noindex and sitemap approach"_): confirm the prop/signature at source before the plan is committed. Sibling to the claims-integrity beat – an externally-anchored _quote_ needs its anchor stated in the spec; an externally-anchored _API_ needs the same, sourced not recalled. **Trigger:** any session touching framework code, authoring a spec that names external API surface, or as a periodic standalone pass
- **Part 7 – cross-diff review on substantive edits (what did the old version carry that the new doesn't?):** Parts 1–6 evaluate the _current state_ of a document. Part 7 evaluates current against _previous_ during transformation. A lockstep pass on `component-reference.md` replaced five "Relaxation candidate:" paragraphs with actual prop signatures. Each new entry was correct, but two pieces of standing guidance baked into the old paragraphs got orphaned – Hero's _"no centred or full-bleed variants"_ and CTA's _"only supports one button – for dual-CTA use Hero"_. The current-state passes couldn't catch them because they scan 'is the new version correct', not 'what did the old carry that the new doesn't'. User prompt: _"Use git to review your net changes and confirm no important detail was lost."_ `git diff HEAD -- [file]` surfaced both immediately. **Principle:** substantive edits (replacing a paragraph with a different _kind_ of paragraph, restructuring, folding bullets) are structurally lossy by default – the new version gets evaluated for fitness, the old's incidental content gets left behind unless deliberately carried over. Run `git diff` before the commit and ask explicitly _"what did the original say that the new doesn't?"_
- **Part 8 – content-routing (when would the next reader look for this?):** Part 8 is about _where_ content lives, not its form. Verification guidance calcified in `active-context.md#Tooling` because that's where it was discovered, but nothing about it was Stitch-specific – it was general-purpose. _"Is active context the right place for this?"_ unlocked the split: directive → `CLAUDE.md` (every session), pattern → `system-patterns.md` (on-demand reference), pointer → `active-context.md`. **The diagnostic question routes it:** every session regardless of task → `CLAUDE.md`; when working the relevant domain → `system-patterns.md`; current focus → `active-context.md`; done-vs-next → `progress.md`. **Principle:** memory bank content accretes in the file where it was discovered (almost always a working-state file), not where it belongs. Promotion to the canonical home has to be deliberate. A rule in `active-context.md#Tooling` is invisible to a session working a different domain even when it applies there too. Add this as a third question to the "review all six files on update" audit, alongside staleness and deduplication
- **Part 9 – polish surfaces standing rules (design-system accumulation):** `/simplify` tightens code (output: code edits). The precise-register polish pass tightens the _design system_ (output: reference-doc rules that compound). Two rules from a homepage polish round, both landed in `component-reference.md#Badge`: **(a) mono vs sans on data surfaces** – mono (Geist Mono) for labels and literal-token values ('code-grade primitive': eyebrows like `META`/`BACKEND`, literal IDs), sans for magnitudes/counts (stats like `600`/`1,000`/`???` read as 'stat', and sans has consistent digit baselines, which fixed a `???`-off-centre quirk mono produced). Don't make all data display mono – the distinction is the signal. **(b) size-appropriate weight + tracking** – tracking goes the _opposite_ direction to instinct: looser at _smaller_ sizes for legibility, not looser at larger for emphasis (my mid-pass bump of global `widest` for mono was backwards – mono is naturally wider and needs less letterspace at small sizes). **Principle:** rules emerging from polish land in the canonical reference doc, not the working-state file – reference doc is what now stands, `progress.md` is what changed
- **Part 10 – instruction-scrubbing (is this an instruction or an explanation?):** A skill carries a cost a reference doc doesn't – its body persists in context for the whole session once loaded, so every line is a recurring token cost. Anthropic's skill-authoring guidance is blunt: _"state what to do rather than narrating how or why."_ The spec-page skill had drifted explanation-heavy, and the narration had crept in through the sections added that same session – the writer narrates the reasoning while it's fresh, the same reflex Part 1 catches as log-style output. The scrub cut decorative why: a practitioner-vs-interviewer analogy, an 'outline is cheaper to redirect than a draft' aside, file-content inventories the agent reads anyway. **The non-obvious half – keep the load-bearing why.** The clause that prevents a predictable misapplication earns its place: the harvest step keeps _"reading old pages earlier anchors you"_, because without it a future run reads them too early and reintroduces the skew. The vault's own convention already draws this line – specs and procedures state what to do, methodologies keep the reasoning that stops misuse. **So the rule isn't 'cut the why', it's 'cut decorative why, keep the one clause that stops a mistake'.** **And test the keeper:** a mistake-stopping clause only earns its place if the surrounding structure doesn't already prevent the mistake – a guard against a path the flow can't take (a 'don't auto-apply' note on a review step whose own _review → ask → apply_ order already makes auto-applying impossible) is the same over-specification as decorative why, dressed as safety, so the keeper is load-bearing only against a mistake the structure leaves reachable. Distinct from Part 1: that separates present from past (reference vs log), this separates instruction from explanation (what vs why). **Generalises** to any always-loaded instruction artefact – skill, CLAUDE.md, system prompt – where the bar for a line is 'does an executor need this to act', not 'is it true'
